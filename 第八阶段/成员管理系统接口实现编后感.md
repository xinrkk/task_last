## 一、项目规划与预期差异

### 最初规划

最初规划时，我设想构建一个功能完善的成员管理系统，包含：

- 完整的用户认证和授权体系
- 管理员和普通用户的双角色权限控制
- 用户注册审核流程
- 完善的RESTful API接口

### 预期与实际的差异

**预期中的理想状态：**

- 代码结构清晰，模块化程度高
- 安全性考虑周全（密码加密、JWT认证、SQL注入防护等）
- 错误处理完善，返回友好的错误信息
- 支持数据库迁移和配置管理

**实际实现的简化版：**

- 将所有代码集中在一个文件中，便于快速理解和演示
- 简化了错误处理，只保留核心逻辑
- 移除了复杂的配置管理，采用硬编码方式



## 二、代码结构设计



```
// 1. 模型层 (Model)
type User struct {
    ID        uint      `gorm:"primaryKey"`
    UserID    string    `gorm:"uniqueIndex"`
    Password  string
    // ... 其他字段
}

// 2. 数据访问层 (通过GORM内联)
db.Where("user_id = ?", userID).First(&user)

// 3. 业务逻辑层 (Service)
func register(c *gin.Context) {
    // 参数验证
    // 业务规则检查
    // 调用数据访问
}

// 4. 控制层 (Handler)
// 实际上register等函数就是handler

// 5. 路由层
r.POST("/register", register)
r.POST("/login", login)
```



### 各模块职责

| 模块         | 职责          | 实现方式          |
| :----------- | :------------ | :---------------- |
| **模型层**   | 定义数据结构  | GORM模型定义      |
| **数据访问** | 数据库CRUD    | GORM内联操作      |
| **业务逻辑** | 实现业务规则  | Handler函数中实现 |
| **认证授权** | JWT生成和验证 | 中间件函数        |
| **路由层**   | API路由分发   | Gin路由组         |

## 三、接口设计思路与困惑

### RESTful API设计

我遵循了RESTful设计原则：



```
GET    /admin/users          # 获取所有成员
GET    /admin/users/:user_id # 获取单个成员
DELETE /admin/users/:user_id # 删除成员
PUT    /admin/users/:user_id/review # 审核用户（非标准REST）
PUT    /profile              # 修改个人信息
```



### 设计上的困惑

**1. 非标准REST操作的处理**



```
// 困惑：审核用户应该用PUT还是PATCH？
PUT /admin/users/:user_id/review
// 或者
PATCH /admin/users/:user_id/status
```



我最终选择了`PUT /review`，因为：

- 语义更清晰，明确表示这是"审核"操作
- 虽然不符合严格的RESTful规范，但更符合业务语义

- 

**3. 状态码的选择**



```
// 困惑：注册成功应该返回201还是200？
c.JSON(http.StatusCreated, ...)  // 201 Created
// 还是
c.JSON(http.StatusOK, ...)       // 200 OK
```



最终使用201表示资源创建成功，更符合HTTP语义。

## 四、开发过程中的最大困难

### 最大的困难：**JWT认证中间件的实现和调试**

**问题描述：**
在实现JWT认证时，遇到了token验证失败的问题，总是返回"登录已过期"。

**定位过程：**

1. **打印日志排查**



```
fmt.Printf("Token string: %s\n", tokenString)
fmt.Printf("Claims: %+v\n", claims)
```



1. **发现问题**：前端传递的token带有"Bearer "前缀，但解析时没有正确处理



```
// 错误的做法
tokenString := c.GetHeader("Authorization")

// 正确的做法
auth := c.GetHeader("Authorization")
tokenString := strings.TrimPrefix(auth, "Bearer ")
```



1. **进一步调试**：发现token签名验证失败



```
// 检查签名密钥是否一致
fmt.Printf("Using secret key: %s\n", jwtKey)
```



1. **最终解决方案**：统一密钥管理和token生成解析逻辑

**查资料过程：**

- 查阅Gin文档关于中间件的使用
- 查看jwt-go库的GitHub issues
- 参考开源项目的JWT实现

**经验总结：**

- 认证中间件是系统的入口，必须保证100%正确
- 详细的日志是定位问题的关键
- 单元测试对这类核心功能至关重要

## 五、项目不足与优化方向

### 当前项目的不足

**1. 安全性问题**



```
// 当前问题：硬编码密钥
var jwtKey = []byte("my_secret_key")

// 应该改进：从环境变量读取
var jwtKey = []byte(os.Getenv("JWT_SECRET"))
```



**2. 错误处理不够完善**



```
// 当前：简单的错误返回
c.JSON(http.StatusInternalServerError, gin.H{"error": "注册失败"})

// 应该：区分错误类型
if errors.Is(err, gorm.ErrDuplicatedKey) {
    // 返回具体错误
}
```

**3. 缺乏输入验证**

```
// 当前：简单的required验证
UserID   string `json:"user_id" binding:"required"`

// 应该：更严格的格式验证
UserID   string `json:"user_id" binding:"required,min=4,max=20,alphanum"`
```

**4. 性能优化缺失**

- 没有使用数据库索引优化查询
- 没有添加缓存层
- 没有分页处理（获取所有用户可能影响性能）
